Integrate SWA with Secrets Manager JWT authentication

\n

This topic describes how to configure federation between Secure Workload Access (SWA) and Secrets Manager by using the JWT authenticator.

\n

After you complete this workflow, a workload can fetch a JWT-SVID from SWA, authenticate to Secrets Manager, and retrieve authorized secrets.

\n

Before you beginCopy bookmark

\n
    \n
  • \n

    Complete Secure Workload Access (SWA) setup and make sure workloads can fetch JWT-SVIDs. For more information, see Get started with SWA on Kubernetes.

    \n
    • \n
  • \n

    Confirm OIDC issuer values. For more information, see Configure OIDC issuer values for SWA integrations.

    \n
    • \n
  • \n

    You must confirm JWT signing and audience requirements when you integrate SWA with Secrets Manager JWT authentication. SWA can issue JWT-SVIDs with RSA or elliptic-curve signing on the trust domain. You must use RSA signing (RS* with RSA_*) with the JWT authenticator. Elliptic-curve signing is not supported on that path. For more information, see Configure JWT requirements for SWA integrations.

    \n
    • \n
  • \n

    Use an account with permissions to manage policies, authenticators, and secrets in Secrets Manager.

    \n
    • \n
  • \n

    Make sure the following tools are available on the host where you run the commands: the Secrets Manager CLI, curl, jq, and the swa-agent binary installed at /opt/swa/bin/swa-agent.

    \n
    • \n
\n \n \n \n \n \n \n \n \n \n
 \n

When SWA attests a workload, it creates a workload host identity under data/swa/trust-domains/<trust-domain>/workloads/. SWA does not create the workloads group. New workload identities have no secret access by default.

\n

You must create the workloads group, annotate the workload host with its SPIFFE ID so the JWT authenticator can map the sub claim, grant the host membership in the group, and grant secret permissions before the workload can retrieve secrets.

\n
\n

\n \n

\n

Configure and load the JWT authenticator policyCopy bookmark

\n

Create a JWT authenticator policy for SWA. The policy defines the JWT authenticator webservice, declares the configuration variables, and sets up groups for workload and operator access.

\n \n \n \n \n \n \n \n \n \n
 
- !policy\n  id: secureWorkloadAccess\n  body:\n  - !webservice\n    annotations:\n      description: JWT authenticator for SWA\n\n  - !variable\n    id: jwks-uri\n\n  - !variable\n    id: token-app-property\n\n  - !variable\n    id: identity-path\n\n  - !variable\n    id: issuer\n\n  - !variable\n    id: audience\n\n  - !group\n    id: apps\n\n  - !permit\n    role: !group apps\n    privilege: [ read, authenticate ]\n    resource: !webservice\n\n  - !webservice status\n\n  - !group\n    id: operators\n\n  - !permit\n    role: !group operators\n    privilege: [ read ]\n    resource: !webservice status
\n
\n

Save the policy as jwt-authenticator.yaml, then load it in the conjur/authn-jwt branch:

\n \n \n \n \n \n \n \n \n \n
 
conjur policy load -b conjur/authn-jwt -f jwt-authenticator.yaml
\n
\n

Set JWT authenticator variablesCopy bookmark

\n

Set the required variables for the secureWorkloadAccess JWT authenticator.

\n

The OIDC discovery document uses the key jwks_uri (with an underscore). The corresponding Secrets Manager variable ID is jwks-uri (with a hyphen).

\n

For SWA, the issuer identifier and JWKS URI are scoped to the trust domain. They follow the pattern https://<tenant>.secretsmgr.cyberark.cloud/api/swa/trust-domains/<trust-domain>. Append /.well-known/jwks for the JWKS URI. The issuer value is the trust domain base URL, not the OIDC discovery document URL.

\n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n \n
\n

Variable

\n 		\n

Description

\n
\n

jwks-uri\n

\n 		\n

Set to the SWA JWKS URL for your trust domain. For Kubernetes, you can read jwks_uri from OIDC discovery (see Get started with SWA on Kubernetes).

\n
\n

token-app-property\n

\n 		\n

Set to sub when the JWT subject claim identifies the workload (SPIFFE ID).

\n
\n

identity-path\n

\n 		\n

Set to data/swa/trust-domains/<trust-domain>/workloads, the branch where SWA registers workload identities. Replace <trust-domain> with your SWA trust domain name.

\n
\n

issuer\n

\n 		\n

Set to the SWA OIDC issuer URL for your trust domain (trust domain base URL). For Kubernetes, you can read issuer from OIDC discovery (see Get started with SWA on Kubernetes).

\n
\n

audience\n

\n 		\n

Set to conjur so it matches the JWT aud claim. If you use a different value in the SWA Server Helm chart for controlPlane.auth.audience, set this variable to that same value.

\n
\n

Run the following commands to set the variables:

\n \n \n \n \n \n \n \n \n \n
 
# JWKS URI — SWA publishes per-trust-domain JWKS at this path\nconjur variable set -i conjur/authn-jwt/secureWorkloadAccess/jwks-uri -v \"https://<tenant>.secretsmgr.cyberark.cloud/api/swa/trust-domains/<trust-domain>/.well-known/jwks\"\n\n# JWT claim used as the application identity (SPIFFE ID)\nconjur variable set -i conjur/authn-jwt/secureWorkloadAccess/token-app-property -v \"sub\"\n\n# Branch where SWA registers workload identities — must match your trust domain\nconjur variable set -i conjur/authn-jwt/secureWorkloadAccess/identity-path -v \"data/swa/trust-domains/<trust-domain>/workloads\"\n\n# Issuer — SWA OIDC issuer is scoped per trust domain\nconjur variable set -i conjur/authn-jwt/secureWorkloadAccess/issuer -v \"https://<tenant>.secretsmgr.cyberark.cloud/api/swa/trust-domains/<trust-domain>\"\n\n# Audience — must match the value configured in the SWA Server Helm chart (controlPlane.auth.audience)\nconjur variable set -i conjur/authn-jwt/secureWorkloadAccess/audience -v \"conjur\"
\n

Enable the JWT authenticatorCopy bookmark

\n

Enable the authn-jwt/secureWorkloadAccess authenticator.

\n \n \n \n \n \n \n \n \n \n
 
conjur authenticator enable --id authn-jwt/secureWorkloadAccess
\n
\n

Register the workload identity and grant authenticator accessCopy bookmark

\n

SWA creates workload host identities under data/swa/trust-domains/<trust-domain>/workloads/ but does not create the workloads group. You must create the group, annotate the workload host with its SPIFFE ID so the JWT authenticator can map the sub claim, and grant the host membership in the group.

\n

Replace <trust-domain> with your SWA trust domain name and <spiffe-id> with the full SPIFFE ID of the workload as issued by SWA (for example, spiffe://example.org/default-nodegroup/ns/my-namespace/sa/my-service-account). When SWA attests a workload, it creates the workload identity in Secrets Manager using the SPIFFE ID as its name.

\n

Save the following policy as workloads-group.yaml:

\n \n \n \n \n \n \n \n \n \n
 
- !group\n  id: workloads\n\n- !host\n  id: workloads/<spiffe-id>\n  annotations:\n    authn-jwt/secureWorkloadAccess/sub: <spiffe-id>\n\n- !grant\n  role: !group workloads\n  member: !host workloads/<spiffe-id>
\n
\n

Load the policy in the trust domain branch:

\n \n \n \n \n \n \n \n \n \n
 
conjur policy load -b data/swa/trust-domains/<trust-domain> -f workloads-group.yaml
\n
\n \n \n \n \n \n \n \n \n \n
 \n

The annotation key must be authn-jwt/secureWorkloadAccess/sub. The annotation value must match the JWT sub claim (SPIFFE ID). If you rename the authenticator policy id in the first section, update the annotation prefix and all conjur/authn-jwt/<id>/... paths consistently.

\n
\n

Save the following policy as authn-jwt-grant.yaml, then load it to grant the workloads group access to the JWT authenticator service:

\n \n \n \n \n \n \n \n \n \n
 
- !grant\n  role: !group apps\n  member: !group /data/swa/trust-domains/<trust-domain>/workloads
\n
\n \n \n \n \n \n \n \n \n \n
 
conjur policy load -f authn-jwt-grant.yaml -b conjur/authn-jwt/secureWorkloadAccess
\n
\n

Create secrets and grant workload permissionsCopy bookmark

\n

Create a secret policy and grant the workload identity permissions to access secrets.

\n

The host path uses an absolute reference (leading /) so it resolves to the identity that SWA creates in the data branch, regardless of which branch you load this policy into.

\n \n \n \n \n \n \n \n \n \n
 
- !policy\n  id: swa/secrets\n  body:\n  - !policy\n    id: myapp\n    body:\n    - !variable\n      id: api-key\n\n    - !permit\n      role: !host /data/swa/trust-domains/<trust-domain>/workloads/<spiffe-id>\n      privilege: [ read, execute ]\n      resource: !variable api-key
\n
\n

Save the policy as secrets-and-permissions.yaml, then load it and set the secret value:

\n \n \n \n \n \n \n \n \n \n
 
conjur policy load -b data -f secrets-and-permissions.yaml\nconjur variable set -i data/swa/secrets/myapp/api-key -v \"sk-1234567890abcdef\"
\n
\n

Test token exchangeCopy bookmark

\n

Fetch a JWT-SVID from SWA, authenticate to Secrets Manager, and retrieve a secret.

\n

Fetch the JWT-SVID from the SWA agent:

\n \n \n \n \n \n \n \n \n \n
 
JWT_TOKEN=$(/opt/swa/bin/swa-agent api fetch jwt \\\n  --audience conjur \\\n  --output json \\\n  --socketPath /run/swa-agent/api.sock | jq -r '.[].svids[]?.svid')
\n
\n

Authenticate to Secrets Manager and retrieve a secret:

\n \n \n \n \n \n \n \n \n \n
 
CONJUR_ACCESS_TOKEN=$(curl -s -X POST \\\n  \"https://<subdomain>.secretsmgr.cyberark.cloud/api/authn-jwt/secureWorkloadAccess/conjur/authenticate\" \\\n  -H \"Content-Type: application/x-www-form-urlencoded\" \\\n  -d \"jwt=$JWT_TOKEN\" | base64 | tr -d '\\n')\n\ncurl -s \\\n  -H \"Authorization: Token token=\\\"$CONJUR_ACCESS_TOKEN\\\"\" \\\n  \"https://<subdomain>.secretsmgr.cyberark.cloud/api/secrets/conjur/variable/data/swa/secrets/myapp/api-key\"
\n

Migrate existing workloads to SPIFFE authenticationCopy bookmark

\n

If you switch an existing workload from a different authentication method to SWA and SPIFFE, you must manually handle the transition to avoid authentication downtime.

\n \n \n \n \n \n \n \n \n \n
 \n

SWA cannot recognize or migrate existing workload identities that you created with non-SPIFFE authenticators. When SWA attests a workload, it creates a new workload identity with a SPIFFE ID.

\n

Delete the existing workload identity, and then grant permissions to the new SPIFFE-based identity. This change causes a brief service interruption.

\n
\n
To migrate an existing workload:
\n
    \n
  1. \n

    Plan the cutover: Identify when your workload will switch authentication methods.

    \n
    2. \n
  2. \n

    Deploy and start SWA agents on your workload nodes. SWA begins attesting the workload and creates a new identity under data/swa/trust-domains/<trust-domain>/workloads/<spiffe-id>.

    \n
    3. \n
  3. \n

    Complete the steps in Register the workload identity and grant authenticator access and Create secrets and grant workload permissions for the new SPIFFE-based identity.

    \n
    4. \n
  4. \n

    Delete the old workload identity from the data branch to prevent access conflicts.

    \n \n \n \n \n \n \n \n \n \n
     
    conjur policy load -b data <<EOF\n- !delete\n  resource: !host /<old-path>/<old-workload-id>\nEOF
    \n
    \n
    5. \n
  5. \n

    Update your workload to use SWA for authentication (fetch SVID from the agent, authenticate using JWT, retrieve secrets).

    \n
    6. \n
  6. \n

    Test and verify the workload retrieves secrets successfully with the new SPIFFE identity.

    \n
    7. \n
