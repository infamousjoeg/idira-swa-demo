Install an SWA agent on a machine

\n

This topic describes how you install a Secure Workload Access (SWA) Agent on a virtual machine (VM) or other machine that runs an SSH server. The SWA Server runs on Kubernetes. You use X.509 proof-of-possession (x509pop) node attestation, the SWA REST API, and Ansible to register the agent with that server.

\n

Before you beginCopy bookmark

\n

Make sure you meet these requirements before you start:

\n
    \n
  • \n

    A trust domain for your SWA deployment (if you need to create one, run only the trust-domain curl in Step 1: Create SWA resources in Get started with SWA on Kubernetes)

    \n
    • \n
  • \n

    A Kubernetes cluster where you deploy the SWA Server

    \n
    • \n
  • \n

    Network connectivity from each agent machine to the SWA Server endpoint

    \n
    • \n
  • \n

    One or more target machines with an SSH server (for example, a Linux VM) and sudo access for the Ansible user

    \n
    • \n
  • \n

    Ansible on a control machine where you run playbooks

    \n
    • \n
  • \n

    Secrets Manager Admin role and an access token for SWA API calls (see Authenticate user)

    \n
    • \n
  • \n

    The swa-agent binary and Ansible roles from your SWA artifact bundle (extract the swa-agent archive from the release folder in the bundle)

    \n
    • \n
\n \n \n \n \n \n \n \n \n \n
 \n

For x509pop design context and how node group names map to certificate Subject CNs, see Design and assign SWA node groups.

\n
\n

Generate the x509pop CA certificateCopy bookmark

\n

Generate a certificate authority (CA) private key and a self-signed CA certificate that you use to attest SWA Agents on your machines.

\n
To generate the CA key and certificate:
\n
    \n
  1. \n

    Generate the CA private key.

    \n \n \n \n \n \n \n \n \n \n
     
    openssl genrsa -out ca.key 4096
    \n
    \n
    2. \n
  2. \n

    Generate the self-signed CA certificate.

    \n \n \n \n \n \n \n \n \n \n
     
    openssl req -new -x509 -days 3650 -key ca.key -out ca.crt \\\n  -subj \"/CN=SWA Node Attestation CA\"
    \n
    \n
    3. \n
\n \n \n \n \n \n \n \n \n \n
 \n

Keep ca.key secure. You use it to sign agent certificates. Use a dedicated intermediate CA for SWA node certificates rather than your organization's root CA.

\n
\n

Configure SWA resources with the REST APICopy bookmark

\n

Create a server group, register and install an SWA Server in that group, then create a node group for your machines. Set shell variables SWA_API_BASE, TOKEN, TRUST_DOMAIN_NAME, SERVER_GROUP_NAME, SERVER_NAME, and NODE_GROUP_NAME as in Get started with SWA on Kubernetes.

\n

Create a server groupCopy bookmark

\n

Embed the CA certificate from the previous section in node_attestation.x509pop.ca_certificates. Format the PEM from ca.crt, then run:

\n \n \n \n \n \n \n \n \n \n
 
export CA_CERT_PEM=$(awk 'NF {sub(/\\r/, \"\"); printf \"%s\\\\n\",$0;}' ca.crt)\n\ncurl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains/${TRUST_DOMAIN_NAME}/server-groups\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d '{\n        \"name\": \"'\"${SERVER_GROUP_NAME}\"'\",\n        \"description\": \"Application server group\",\n        \"node_attestation\": {\n            \"x509pop\": {\n                \"ca_certificates\": \"'\"${CA_CERT_PEM}\"'\"\n            }\n        }\n    }'
\n
\n

For all available fields and full response details, see Create a server group.

\n

Save the server group name. You use it when you register the SWA Server.

\n

Register and install the SWA ServerCopy bookmark

\n

Follow Get started with SWA on Kubernetes to register and install the SWA Server in SERVER_GROUP_NAME. Use Step 2 and the SWA Server install in Step 3 only. Do not run the server group or node group commands in Step 1 of that topic. Skip the Kubernetes node group and SWA Agent Helm steps in that topic.

\n

Create a node groupCopy bookmark

\n

After the SWA Server is running, associate a node group with the server group and set workload_type to unix for machine workloads.

\n \n \n \n \n \n \n \n \n \n
 
curl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains/${TRUST_DOMAIN_NAME}/server-groups/${SERVER_GROUP_NAME}/node-groups\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d '{\n        \"name\": \"'\"${NODE_GROUP_NAME}\"'\",\n        \"description\": \"Unix workload node group with defaults\",\n        \"workload_type\": \"unix\"\n    }'
\n
\n

For all available fields and full response details, see Create node group.

\n

The Subject CN in each agent certificate must match the node group name exactly (for example, unix-nodegroup-1). The SWA Server uses that value during x509pop attestation to assign the node to the node group.

\n

Install the SWA Agent with AnsibleCopy bookmark

\n

Use Ansible to copy agent certificates to each machine and install the SWA Agent binary.

\n

Define the inventoryCopy bookmark

\n

Create an inventory file that points to your target machines.

\n \n \n \n \n \n \n \n \n \n
 
all:\n  children:\n    nodes:\n      hosts:\n        my-vm:\n          ansible_host: <vm-ip>\n          ansible_user: <user>
\n
\n

Generate an x509pop certificate per agentCopy bookmark

\n

For each agent, generate a key pair and a certificate signed by the CA. Set the CSR Subject CN to NODE_GROUP_NAME (the node group you created earlier).

\n \n \n \n \n \n \n \n \n \n
 
# Generate agent private key\nopenssl genrsa -out agent.key 2048\n\n# Generate CSR (CN must match NODE_GROUP_NAME)\nopenssl req -new -key agent.key -out agent.csr \\\n  -subj \"/CN=${NODE_GROUP_NAME}\"\n\n# Sign with the CA\nopenssl x509 -req -days 365 -in agent.csr \\\n  -CA ca.crt -CAkey ca.key -CAcreateserial -out agent.crt
\n
\n

You can run these commands as tasks in your playbook before the roles copy certificates to each host.

\n

Run the Ansible playbookCopy bookmark

\n

Use the copyx509pop-agent and swa-agent roles from your artifact bundle. Set swa_agent_servers to the address agents use to reach the SWA Server and bundle_source_url to your tenant SWA CA bundle endpoint.

\n \n \n \n \n \n \n \n \n \n
 
---\n- name: Install SWA Agent\n  hosts: nodes\n  gather_facts: true\n  become: true\n  vars:\n    swa_agent_log_level: \"debug\"\n    swa_agent_binary: \"<path-to-swa-agent-binary-from-artifact-bundle>\"\n    node_attestation_type: x509pop\n    swa_agent_x509pop_cert: \"/etc/swa/x509pop.cert\"\n    swa_agent_x509pop_key: \"/etc/swa/x509pop.key\"\n    swa_agent_servers:\n      - addr: \"<IP/host for the server>:8443\"\n    swa_trust_domain: \"<trust-domain-name>\"\n    bundle_source_url: \"https://<tenant-base-url>/api/swa/trust-domains/{{ swa_trust_domain }}/.well-known/ca-bundles\"\n  roles:\n    - copyx509pop-agent\n    - swa-agent
\n
\n
To install agents with Ansible:
\n
    \n
  1. \n

    Save the playbook and inventory files on your control machine.

    \n
    2. \n
  2. \n

    Run the playbook against your inventory.

    \n \n \n \n \n \n \n \n \n \n
     
    ansible-playbook -i inventory.yml swa-playbook.yml
    \n
    \n
    3. \n
\n

For Unix workload attestor defaults in the Ansible role, see SPIFFE templates, attestors, and registration policies for SWA.

\n

Verify the installationCopy bookmark

\n

Confirm that the SWA Agent service is running and can reach the SWA Server.

\n
To verify agent status on a target machine:
\n
    \n
  1. \n

    Check the agent service status.

    \n \n \n \n \n \n \n \n \n \n
     
    sudo systemctl status swa-agent\nsudo journalctl -u swa-agent -f\nls -la /run/swa-agent/api.sock
    \n
    \n
    2. \n
  2. \n

    Optionally, fetch a JWT SVID from the agent Workload API socket.

    \n \n \n \n \n \n \n \n \n \n
     
    /opt/swa/bin/swa-agent api fetch jwt \\\n  --audience conjur \\\n  --socketPath /run/swa-agent/api.sock
    \n
    \n
    3. \n
\n

From the Ansible control machine, you can run ad hoc checks instead of SSH to each host:

\n \n \n \n \n \n \n \n \n \n
 
ansible my-vm -i inventory.yml -a \"sudo systemctl status swa-agent\"\nansible my-vm -i inventory.yml -a \"ls -la /run/swa-agent/api.sock\"
\n
