Install SWA on Kubernetes with Helm

\n

This topic describes how you install the Secure Workload Access (SWA) Server and SWA Agents on Kubernetes by using Helm charts.

\n

Before you beginCopy bookmark

\n

Make sure you meet these requirements before you start:

\n
    \n
  • \n

    A Kubernetes cluster for the SWA Server deployment

    \n
    • \n
  • \n

    Helm 3 installed on the machine where you run deployment commands

    \n
    • \n
  • \n

    The SWA Server and SWA Agent Helm charts are available on the installation machine

    \n
    • \n
  • \n

    SWA container images available in a container registry your cluster can reach

    \n
    • \n
  • \n

    An authn_id value from the SWA Server registration. For details, see Register an SWA server.

    \n
    • \n
\n \n \n \n \n \n \n \n \n \n
 \n

For Kubernetes deployments, use Helm as the primary installation method for the SWA Server and SWA Agents.

\n
\n

Install the SWA ServerCopy bookmark

\n

Install the SWA Server chart first so agents can connect to the server endpoint.

\n
To install the SWA Server chart:
\n
    \n
  1. \n

    Run Helm install with your Secrets Manager - SaaS API endpoint and the SWA Server authn_id.

    \n

    Set controlPlane.url to your tenant base URL, for example https://<subdomain>.secretsmgr.cyberark.cloud. Set controlPlane.auth.loginURL to the authn_id value from the register-server response (the chart field name is loginURL).

    \n \n \n \n \n \n \n \n \n \n
     
    helm install swa-server ./swa-server \\\n  --namespace swa-system \\\n  --create-namespace \\\n  --set controlPlane.url=https://<subdomain>.secretsmgr.cyberark.cloud \\\n  --set controlPlane.auth.loginURL=\"<authn_id captured from SWA Server creation>\" \\\n  --set rbac.createTokenReviewRole=true \\\n  --set trustDomain.name=${TRUST_DOMAIN_NAME} \\\n  --set image.repository=<container-repository-url>/swa-server \\\n  --set image.tag=latest
    \n
    \n
    2. \n
  2. \n

    Confirm that the SWA Server pod is running in the swa-system namespace.

    \n
    3. \n
\n

Install SWA AgentsCopy bookmark

\n

Install SWA Agents as a DaemonSet so each node can provide workload identity services locally.

\n
To install the SWA Agent chart:
\n
    \n
  1. \n

    Run Helm install for the SWA Agent chart.

    \n \n \n \n \n \n \n \n \n \n
     
    helm install swa-agent ./swa-agent \\\n  --namespace swa-system \\\n  --set trustDomain.name=${TRUST_DOMAIN_NAME} \\\n  --set server.address=swa-server.swa-system.svc.cluster.local:8443 \\\n  --set nodeAttestor.k8s_psat.cluster=prod-cluster \\\n  --set image.repository=<container-repository-url>/swa-agent \\\n  --set image.tag=latest \\\n  --set podLabels.swa_nodegroup=${NODE_GROUP_NAME}
    \n
    \n
    2. \n
  2. \n

    Confirm that the SWA Agent pods are running on your workload nodes.

    \n
    3. \n

Why SWA uses separate chartsCopy bookmark

\n

The SWA Server and the SWA Agent use separate charts because they have different deployment patterns, lifecycle requirements, and configuration inputs.

\n
    \n
  • \n

    The SWA Server runs as a Deployment and can scale independently.

    \n
    • \n
  • \n

    The SWA Agent runs as a DaemonSet and serves workloads on each node.

    \n
    • \n
  • \n

    The SWA Server requires control plane settings, while the SWA Agent requires server endpoint and node attestor settings.

    \n
    • \n
\n

Key parametersCopy bookmark

\n

Set these parameters explicitly in your values or command-line overrides:

\n
    \n
  • \n

    Server chart: controlPlane.url, controlPlane.auth.loginURL, rbac.createTokenReviewRole, trustDomain.name, image.repository, and image.tag

    \n
    • \n
  • \n

    Agent chart: trustDomain.name, server.address, nodeAttestor.k8s_psat.cluster, image.repository, image.tag, and podLabels.swa_nodegroup

    \n
    • \n
\n

The podLabels.swa_nodegroup value must match your node group name and server group attestation settings. For design guidance, see Design and assign SWA node groups.

\n

Node attestor optionsCopy bookmark

\n

For Kubernetes environments, use k8s_psat.

\n \n \n \n \n \n \n \n \n \n
 
nodeAttestor:\n  type: k8s_psat\n  k8s_psat:\n    cluster: <cluster-name>\n    tokenPath: /var/run/secrets/kubernetes.io/serviceaccount/token
\n
\n

Use x509pop only when your environment requires certificate-based node attestation. To install SWA Agents on virtual machines or other machines outside the cluster, see Install an SWA agent on a machine.

\n \n \n \n \n \n \n \n \n \n
 
helm install swa-agent ./swa-agent \\\n  --namespace swa-system \\\n  --set server.address=swa-server:8443 \\\n  --set nodeAttestor.type=x509pop \\\n  --set-file nodeAttestor.x509pop.cert=/path/to/agent.crt \\\n  --set-file nodeAttestor.x509pop.key=/path/to/agent.key
\n
\n

Workload attestorsCopy bookmark

\n

The SWA Agent can run multiple workload attestors. The common Kubernetes pattern uses both unix and k8s.

\n \n \n \n \n \n \n \n \n \n
 
workloadAttestors:\n  - type: unix\n  - type: k8s\n    config:\n      nodeNameEnv: MY_NODE_NAME
\n
\n

If the k8s workload attestor reports kubelet certificate errors, especially on self-managed Kubernetes clusters, see Troubleshoot Secure Workload Access (SWA).

\n

Verify the deploymentCopy bookmark

\n

Verify that both the SWA Server and the SWA Agent resources are healthy after installation.

\n
To verify Helm deployment status:
\n
    \n
  1. \n

    Check Helm release status.

    \n \n \n \n \n \n \n \n \n \n
     
    helm list -n swa-system
    \n
    \n
    2. \n
  2. \n

    Check pod status.

    \n \n \n \n \n \n \n \n \n \n
     
    kubectl get pods -n swa-system
    \n
    \n
    3. \n
\n

For the end-to-end setup flow, see Get started with SWA on Kubernetes.
