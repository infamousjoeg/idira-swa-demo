Get started with SWA on Kubernetes

\n

This topic describes how you set up Secrets Manager Secure Workload Access (SWA) on Kubernetes to manage SPIFFE-based workload identities and fetch secrets from Secrets Manager.

\n

With SWA, workloads are attested and issued a unique SPIFFE Verifiable Identity Document (SVID). Workloads use the SVID to authenticate and retrieve secrets.

Set up Secure Workload AccessCopy bookmark

\n

Use this workflow to set up Secure Workload Access (SWA) and enable workloads to fetch secrets.

\n

Prepare API authenticationCopy bookmark

\n

Get a Secrets Manager access token and set shell variables before you run SWA API commands.

\n
To prepare API access:
\n
    \n
  1. \n

    Get a Secrets Manager access token for API calls by using an account with the Secrets Manager Admin role. For the full authentication flow, see Authenticate user.

    \n
    2. \n
  2. \n

    Optional: set these shell variables to reuse values across the commands in this workflow. If you do not use shell variables, replace each variable reference directly in the commands.

    \n

    The examples use SWA_API_BASE for the tenant host. Set it from your tenant subdomain. The host matches https://<subdomain>.secretsmgr.cyberark.cloud.

    \n \n \n \n \n \n \n \n \n \n
     
    export TENANT_SUBDOMAIN=\"<subdomain>\"\nexport SWA_API_BASE=\"https://${TENANT_SUBDOMAIN}.secretsmgr.cyberark.cloud\"\nexport TOKEN=\"<token>\"\nexport TRUST_DOMAIN_NAME=\"prod.example\"\nexport SERVER_GROUP_NAME=\"k8s-prod\"\nexport NODE_GROUP_NAME=\"k8s-prod-ng\"\nexport SERVER_NAME=\"swa-server-a\"

Step 1: Create SWA resourcesCopy bookmark

\n

Create the SPIFFE hierarchy through the API. Resource creation also creates system-managed branches in Secrets Manager, such as data/swa.

\n

To manage trust domains, server groups, node groups, and related resources with Terraform instead of the REST API, install the SWA Terraform provider. For details, see Install the SWA Terraform provider.

\n
API conventions for SWA REST calls\n
\n \n \n \n \n \n \n \n \n \n
 \n

API conventions: Use the SWA API base host https://<subdomain>.secretsmgr.cyberark.cloud. SWA endpoints use the /api/swa prefix. Authenticate requests with Authorization: Token token=\"<access token from authenticate endpoint>\". Use Accept: application/x.secretsmgr.v2+json for v2 APIs. Do not use a Beta header. URL-encode any resource name that appears in a path segment.

\n
\n
\n
\n \n \n \n \n \n \n \n \n \n
 \n

Use the commands in this workflow for a minimal setup path. For complete field definitions, optional parameters, validation rules, and response details, see Secure Workload Access APIs.

\n

For node group design (attestation, swa_nodegroup, identity boundaries), see Design and assign SWA node groups. For SPIFFE ID templates, attestors, and workload registration policies, see SPIFFE templates, attestors, and registration policies for SWA.

\n
\n
To create SWA resources:
\n
    \n
  1. \n

    Create a trust domain: Define the root SPIFFE namespace.

    \n

    Use this command:

    \n \n \n \n \n \n \n \n \n \n
     
    curl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d '{\n        \"name\": \"'\"${TRUST_DOMAIN_NAME}\"'\",\n        \"jwt\": {\n            \"signature_algorithm\": \"RS512\",\n            \"signing_key_ttl\": 86400,\n            \"signing_key_type\": \"RSA_4096\",\n            \"token_ttl\": 300\n        },\n        \"x509\": {\n            \"workload_ttl\": 3600\n        }\n    }'
    \n
    \n \n \n \n \n \n \n \n \n \n
     \n

    New trust domains default to RSA JWT signing with RS512 and RSA_4096, which works with Secrets Manager JWT authentication and Secrets Manager - SaaS for JWT-SVID flows. Elliptic-curve signing is also supported on the trust domain. This example sets those values explicitly; you can omit the jwt object to use the same defaults. For integration-specific signing requirements, see Configure JWT requirements for SWA integrations.

    \n
    \n

    For all available fields and full response details, see Register a trust domain.

    \n

    Save the trust domain name. Depending on the final authentication configuration, you might also need jwt.discovery_endpoints.oidc_discovery_url and jwt.discovery_endpoints.jwks_uri for later steps.

    \n
    2. \n
  2. \n

    Create a server group: Define a logical group for SWA Servers. Use the trust domain name from the previous step.

    \n

    Use this command:

    \n \n \n \n \n \n \n \n \n \n
     
    curl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains/${TRUST_DOMAIN_NAME}/server-groups\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d '{\n        \"name\": \"'\"${SERVER_GROUP_NAME}\"'\",\n        \"description\": \"Kubernetes manual test server group\",\n        \"node_attestation\": {\n            \"k8s_psat\": {\n                \"clusters\": {\n                    \"prod-cluster\": {\n                        \"service_account_allow_list\": [\n                            \"swa-system:swa-agent\"\n                        ],\n                        \"audience\": [\n                            \"swa-server\"\n                        ],\n                        \"allowed_pod_label_keys\": [\n                            \"swa_nodegroup\"\n                        ]\n                    }\n                }\n            }\n        }\n    }'
    \n
    \n

    For all available fields and full response details, see Create a server group.

    \n

    Save the server group name. You use it when you create node groups and register the SWA Server.

    \n
    3. \n
  3. \n

    Create a node group: Define attestation rules that control which workloads can get an SVID. Use the trust domain and server group name values from the previous steps.

    \n

    Use this command:

    \n \n \n \n \n \n \n \n \n \n
     
    curl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains/${TRUST_DOMAIN_NAME}/server-groups/${SERVER_GROUP_NAME}/node-groups\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d '{\n        \"name\": \"'\"${NODE_GROUP_NAME}\"'\",\n        \"description\": \"Kubernetes production node group\",\n        \"workload_type\": \"kubernetes\"\n    }'
    \n
    \n

    For all available fields and full response details, see Create node group.

    \n
    4. \n

Step 2: Register the SWA ServerCopy bookmark

\n

Register the SWA Server to get installation credentials; the request includes JWT authenticator data so Secrets Manager can validate service account tokens from your cluster (for field definitions, see Create authenticator and Register an SWA server).

\n

SWA uses a Kubernetes service account to authenticate the SWA Server to Secrets Manager - SaaS so the server can retrieve its configuration; this JWT authentication is used only by the SWA Server and not by workloads.

\n
To register a server:
\n

After discovery, complete the step that matches your tenant network path. You register the server once, using either jwks_uri or public_keys.

\n
    \n
  1. \n

    Discover the Kubernetes API server OpenID configuration from a context that can reach your cluster. Run the following command:

    \n \n \n \n \n \n \n \n \n \n
     
    kubectl get --raw '/.well-known/openid-configuration'
    \n
    \n

    The response is JSON. Save the issuer value and the jwks_uri value. You use them in the registration request when the JWKS URL is reachable from your Secrets Manager - SaaS tenant, as described in the next step.

    \n
    2. \n
  2. \n

    JWKS URL is reachable from the tenant: Send a POST request to register the server. Set authentication.data with issuer and jwks_uri. Do not set public_keys.

    \n

    Set issuer and jwks_uri from the discovery output. Your operations team might publish the same issuer and JWKS on publicly routable URLs. If they do, use those URLs so Secrets Manager can reach them.

    \n

    Use a request body similar to the following example:

    \n \n \n \n \n \n \n \n \n \n
     
    curl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains/${TRUST_DOMAIN_NAME}/server-groups/${SERVER_GROUP_NAME}/components\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d '{\n        \"name\": \"'\"${SERVER_NAME}\"'\",\n        \"authentication\": {\n            \"type\": \"JWT\",\n            \"data\": {\n                \"sub\": \"system:serviceaccount:swa-system:swa-server\",\n                \"issuer\": \"'\"${ISSUER}\"'\",\n                \"jwks_uri\": \"'\"${JWKS_URI}\"'\",\n                \"audience\": \"conjur\"\n            }\n        }\n    }'
    \n
    \n

    Set shell variables ISSUER and JWKS_URI to the issuer and jwks_uri values from your discovery output. Those strings are defined by your cluster; do not substitute documentation project variables (for example https://<subdomain>.secretsmgr.cyberark.cloud) for them. Use conjur for audience so it matches the JWT aud claim your Kubernetes service account tokens use for this flow. For details, see Create authenticator.

    \n
    3. \n
  3. \n

    JWKS URL is not reachable from the tenant: Send a POST request to register the server. Set authentication.data with issuer and public_keys. Omit jwks_uri.

    \n

    issuer is required when you use public_keys. Use the public_keys object shape in Create authenticator. Obtain the JWKS JSON from the cluster. For example, run kubectl get --raw against the jwks_uri path from the discovery document. If jwks_uri is a relative path on the API server, pass that path to kubectl get --raw.

    \n

    Build a JSON file for the request body. The authentication.data object should look like the following example. Replace the empty keys array with the keys array from the JWKS document you obtained with kubectl get --raw (the JWKS document is the JSON object whose top-level property is keys).

    \n \n \n \n \n \n \n \n \n \n
     
    {\n    \"name\": \"<server-name>\",\n    \"authentication\": {\n        \"type\": \"JWT\",\n        \"data\": {\n            \"sub\": \"system:serviceaccount:swa-system:swa-server\",\n            \"issuer\": \"<issuer from discovery>\",\n            \"audience\": \"conjur\",\n            \"public_keys\": {\n                \"type\": \"jwks\",\n                \"value\": {\n                    \"keys\": []\n                }\n            }\n        }\n    }\n}
    \n
    \n

    Send the file with curl, for example:

    \n \n \n \n \n \n \n \n \n \n
     
    curl -sS -X POST \"${SWA_API_BASE}/api/swa/trust-domains/${TRUST_DOMAIN_NAME}/server-groups/${SERVER_GROUP_NAME}/components\" \\\n    -H \"Authorization: Token token=\\\"${TOKEN}\\\"\" \\\n    -H \"Accept: application/x.secretsmgr.v2+json\" \\\n    -H \"Content-Type: application/json\" \\\n    -d @register-swa-server.json
    \n
    \n

    For a complete public_keys example, see Create authenticator.

    \n
    4. \n
\n

For advanced JWT authenticator fields and full response details, see Register an SWA server.

\n
    \n
  1. \n

    Save the authn_id value from the response. Use it when you install the SWA Server.

Step 3: Install SWA componentsCopy bookmark

\n

Install the SWA Server and SWA Agents using supported deployment methods.

\n
To install SWA:
\n
    \n
  1. \n

    Download SWA artifacts from the CyberArk Marketplace under CyberArk solutions.

    \n

    Make sure you have both artifact types:

    \n
      \n
    • \n

      Container images for SWA components

      \n
      • \n
    • \n

      Helm chart package for the SWA Server deployment

      \n
      • \n
    \n

    If container images are provided as .tar.gz archives, extract and import them into a container registry that your Kubernetes cluster can reach.

    \n

    Store the Helm chart on the installation machine where you run Helm commands.

    \n

    For a Helm-based installation workflow and command examples, see Install SWA on Kubernetes with Helm.

    \n
    2. \n
  2. \n

    Install the SWA Server: Deploy the SWA Server to your Kubernetes cluster and provide the authn_id.

    \n

    Set controlPlane.auth.loginURL to the authn_id value from the register-server response (the chart field name is loginURL).

    \n

    For Kubernetes deployments, use Helm as the primary installation method for the SWA Server.

    \n \n \n \n \n \n \n \n \n \n
     
    helm install swa-server ./swa-server \\\n  --namespace swa-system \\\n  --create-namespace \\\n  --set controlPlane.url=https://<subdomain>.secretsmgr.cyberark.cloud \\\n  --set controlPlane.auth.loginURL=\"<authn_id captured from SWA Server creation>\" \\\n  --set rbac.createTokenReviewRole=true \\\n  --set trustDomain.name=${TRUST_DOMAIN_NAME} \\\n  --set image.repository=<container-repository-url>/swa-server \\\n  --set image.tag=latest
    \n
    \n
    3. \n
  3. \n

    Install SWA Agents: Deploy agents on nodes that run workloads.

    \n

    For Kubernetes deployments, install SWA Agents by using the SWA Agent Helm chart as a DaemonSet. For virtual machines or other machines outside the cluster, see Install an SWA agent on a machine.

    \n \n \n \n \n \n \n \n \n \n
     
    helm install swa-agent ./swa-agent \\\n  --namespace swa-system \\\n  --set trustDomain.name=${TRUST_DOMAIN_NAME} \\\n  --set server.address=swa-server.swa-system.svc.cluster.local:8443 \\\n  --set nodeAttestor.k8s_psat.cluster=prod-cluster \\\n  --set image.repository=<container-repository-url>/swa-agent \\\n  --set image.tag=latest \\\n  --set podLabels.swa_nodegroup=${NODE_GROUP_NAME}
    \n
    \n
    4. \n
  4. \n

    Confirm installation status between the SWA Agents and the SWA Server by running this command:

    \n \n \n \n \n \n \n \n \n \n
     
    kubectl get pods -n swa-system
    \n
    \n

    This command verifies that swa-server and swa-agent started correctly. For deeper debugging, check pod logs by running:

    \n \n \n \n \n \n \n \n \n \n
     
    kubectl logs <pod-name> -n swa-system
    \n
    \n
    5. \n

Step 4: Get an SVID from the agentCopy bookmark

\n

Workloads call the SWA Agent's Workload API through a local socket to get identities.

\n
To generate an SVID:
\n
    \n
  1. \n

    Connect the workload to the agent Workload API socket. Depending on your installation, the socket can be at a path such as /run/swa-agent/api.sock, or under another directory on the node (for example when the agent exposes a socket under /tmp/swa-agent).

    \n

    The following example pod mounts the host directory that contains the agent socket and runs swa-agent api fetch jwt to retrieve a JWT SVID. Replace <change-this-to-your-repo> in the image value with your container registry and image reference.

    \n \n \n \n \n \n \n \n \n \n
     
    apiVersion: v1\nkind: Pod\nmetadata:\n  name: svid-fetcher\n  namespace: default\n  labels:\n    app.kubernetes.io/name: svid-fetcher\nspec:\n  restartPolicy: Never\n  containers:\n    - name: svid-fetcher\n      image: <change-this-to-your-repo>/swa-agent:latest\n      args:\n        - api\n        - fetch\n        - jwt\n        - --audience=conjur\n        - --socketPath=/tmp/swa-agent/public/api.sock\n        - --output=json\n      volumeMounts:\n        - name: agent-socket\n          mountPath: /tmp/swa-agent\n  volumes:\n    - name: agent-socket\n      hostPath:\n        path: /tmp/swa-agent\n        type: Directory
    \n
    \n
    2. \n
  2. \n

    Save the manifest to a YAML file, apply it to your cluster, then view the pod logs after the pod completes. The log output reveals the JWT SVID token for the pod.

    \n \n \n \n \n \n \n \n \n \n
     
    kubectl apply -f svid-fetcher.yaml\nkubectl logs -n default svid-fetcher
    \n
    \n
    3. \n

Step 5: Authorize workloads to pull secrets from Secrets ManagerCopy bookmark

\n

Attested workloads appear in the Secrets Manager inventory under data/swa/trust-domains/<trust-domain-name>/workloads/, where <trust-domain-name> matches your TRUST_DOMAIN_NAME (for example, prod.example in this topic). Authorize workloads before they can access secrets.

\n

For the complete workflow (JWT authenticator policy and variables, enablement, and secret grants), see Integrate SWA with Secrets Manager JWT authentication.

\n

For more information about supported signing algorithms and audience values, see Configure JWT requirements for SWA integrations.

\n

Verify successCopy bookmark

\n

Use these checks to confirm Secure Workload Access (SWA) installation and connectivity.

\n
    \n
  • \n

    Agent socket: Confirm the Workload API socket exists where your deployment exposes it, for example ls -la /run/swa-agent/api.sock, or the path under /tmp/swa-agent if that matches your node layout (see Step 4: Get an SVID from the agent).

    \n
    • \n
  • \n

    JWT fetch test: Run swa-agent api fetch jwt --audience conjur --output json --socketPath <path-to-socket>, using the socket path for your environment (for example /run/swa-agent/api.sock or /tmp/swa-agent/public/api.sock). Alternatively, apply the example pod from Step 4: Get an SVID from the agent and run kubectl logs -n default svid-fetcher to read the JWT SVID from the pod output.

    \n
    • \n
  • \n

    Server pods: kubectl get pods --all-namespaces | grep swa-server

    \n
    • \n
  • \n

    Secret retrieval: After JWT authentication and policy are configured, confirm token exchange and secret access using the checks in Integrate SWA with Secrets Manager JWT authentication.
